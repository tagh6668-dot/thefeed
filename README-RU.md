# thefeed

**Читалка лент и лёгкий мессенджер поверх DNS** для сетей, где проходят только DNS-запросы. Читайте Telegram-каналы и публичные аккаунты X и обменивайтесь сквозно-зашифрованными сообщениями с другими пользователями — всё через обычный DNS.

## Установка приложения

*Для тех, кто просто хочет читать ленты и переписываться — сервер не нужен, достаточно импортировать конфиг в клиент.*

Скачайте клиент для своей платформы из последнего релиза — выберите доступное зеркало: **[GitHub](https://github.com/sartoopjj/thefeed/releases/latest)** · **[GitLab](https://gitlab.com/sartoopjj/thefeed/-/releases)**.

| Платформа | Примечание |
|-----------|------------|
| **Android** (7.0+) | APK. Выберите `arm64-v8a` (телефоны примерно с 2017 г.) или `armeabi-v7a` (только старые 32-битные). Неверная архитектура установится, но не запустится. |
| **iOS** (13+) | Установка через **[TestFlight](https://testflight.apple.com/join/J6bfxDdZ)**. Сборка для App Store в планах; также можно собрать из исходников в [ios/](ios/) — см. [Сборка из исходников](#build). |
| **Windows** (10/11) | `.exe` **не подписан**, поэтому SmartScreen покажет *«Windows protected your PC»*, а Defender может поместить его в карантин — это известное **ложное срабатывание** для DNS-туннелей, **не вирус**. Нажмите **More info → Run anyway**; если удалён — восстановите из Defender → *Protection history*; при сомнениях сверьте SHA-256 со страницы релиза. |
| **macOS** | Универсальный `.dmg` (Intel + Apple Silicon), устанавливает `Thefeed.app` перетаскиванием. Не подписан: при первом запуске правый клик → **Open** или `xattr -dr com.apple.quarantine /Applications/Thefeed.app`. |
| **Linux / Termux** | Бинарник `thefeed-client` — запустите и откройте `http://127.0.0.1:8080`. |

Затем откройте **Настройки → Конфиги** и импортируйте конфиг (или введите домен + пароль). DNS-резолверы управляются во вкладке **Резолвер** — общий банк для всех конфигов плюс сканер для поиска новых.

**Публичные конфиги для теста:** [@thefeedconfig](https://t.me/thefeedconfig).

<a id="run-server"></a>

## Запуск сервера

*Для операторов, которые хостят ленты для других.* Сервер работает **вне** цензурируемой сети, тянет данные из Telegram / X и отвечает на зашифрованные DNS-запросы. Настройка в два шага: **(1) DNS-записи**, затем **(2) установка**.

### 1. DNS-записи

Нужны одна запись **A** и одно делегирование **NS**. Пусть IP сервера — `203.0.113.10`, домен — `example.com`.

| # | Type | Name | Value | Назначение |
|---|------|------|-------|------------|
| 1 | A  | `ns.example.com` | `203.0.113.10`   | Указывает имя хоста на ваш сервер |
| 2 | NS | `t.example.com`  | `ns.example.com` | Делегирует поддомен **ленты** вашему серверу |
| 3 *(опц.)* | NS | `c.example.com` | `ns.example.com` | Делегирует поддомен **мессенджера** — только если включаете [чат](#messenger) |

Записи **1–2** обязательны для ленты. Запись **3** нужна только для необязательного [мессенджера](#messenger), который должен использовать **отдельный** от ленты поддомен (например `c.example.com`).

### 2. Установка сервера

Когда DNS готов, установите скриптом или через Docker.

#### Вариант A — скрипт установки (Linux + systemd)

Скрипт сам выбирает доступное зеркало (сначала GitHub, потом GitLab); `--gitlab` форсирует GitLab.

```bash
# Зеркало GitHub
sudo bash -c "$(curl -Ls https://raw.githubusercontent.com/sartoopjj/thefeed/main/scripts/install.sh)"

# Зеркало GitLab (когда аккаунт GitHub недоступен)
sudo bash -c "$(curl -Ls https://gitlab.com/sartoopjj/thefeed/-/raw/main/scripts/install.sh)" -- --gitlab
```

Скрипт скачает свежий бинарник, спросит домен / пароль / каналы / аккаунты X, спросит про логин в Telegram (рекомендуется **Нет** — публичные каналы работают без него) и настроит systemd-сервис. Запускайте повторно для **обновления**.

Прочие действия (передайте скрипт в `sudo bash -s -- <flag>`):

| Flag | Действие |
|------|----------|
| `--version v0.9.2` (или `-v`) | Установить конкретный тег (откат) |
| `--pre` | Установить свежайший пре-релиз (beta / rc) |
| `--list` | Список последних релизов |
| `--login` | Повторный вход в Telegram |
| `--config` | Печать URI импорта (домен, ключ, ключ сервера `sk=`, стартовые резолверы) |
| `--uninstall` | Удалить сервис |

#### Вариант B — Docker

Инструменты Go не нужны. Базовый образ `alpine:3.21` (~23 МБ), работает от non-root пользователя `thefeed` (UID 1000).

```bash
# 1. Настройка — задайте THEFEED_DOMAIN и THEFEED_KEY (раскомментируйте Telegram-переменные для приватных каналов)
cp .env.example .env && nano .env

# 2. Добавьте каналы
mkdir -p data
cp configs/channels.txt data/
cp configs/x_accounts.txt data/   # опционально

# 3. Собрать и запустить (внутри контейнера слушает :5300/udp)
docker compose up -d
docker compose logs -f

# 4. Напечатать клиентский конфиг импорта (URI thefeed:// — домен, ключ, ключ сервера sk=, резолверы),
#    чтобы выдать пользователям, как это делает скрипт в конце:
docker compose run --rm server --print-config --data-dir /data --domain YOUR_DOMAIN --key YOUR_KEY
```

Затем настройте [перенаправление порта 53](#port53) ниже. Для приватных каналов сначала выполните одноразовый интерактивный вход:

```bash
docker compose run -it --rm server --login-only --data-dir /data \
  --domain YOUR_DOMAIN --key YOUR_KEY \
  --api-id YOUR_API_ID --api-hash YOUR_HASH --phone YOUR_PHONE
# затем уберите --no-telegram из docker-compose.yml, добавьте Telegram-флаги и `docker compose up -d`
```

<a id="port53"></a>

### Порт 53

Сервер должен принимать **внешний** DNS на UDP **53**, но привязка к `:53` напрямую конфликтует с `systemd-resolved`. Поэтому он слушает непривилегированный порт (`:5300`), а вы перенаправляете внешний `:53` на него через `iptables`. Локальный DNS хоста продолжает работать — перенаправляются только пакеты с внешнего интерфейса.

```bash
# Замените eth0 на ваш интерфейс (смотрите: ip a)
sudo iptables  -I INPUT -p udp --dport 5300 -j ACCEPT
sudo iptables  -t nat -I PREROUTING -i eth0 -p udp --dport 53 -j REDIRECT --to-ports 5300
sudo ip6tables -I INPUT -p udp --dport 5300 -j ACCEPT
sudo ip6tables -t nat -I PREROUTING -i eth0 -p udp --dport 53 -j REDIRECT --to-ports 5300

# Сохранить после перезагрузки (Debian/Ubuntu)
sudo apt install -y iptables-persistent && sudo netfilter-persistent save
```

Если что-то сломалось — **мгновенно откатите**:

```bash
sudo iptables -t nat -D PREROUTING -i eth0 -p udp --dport 53 -j REDIRECT --to-ports 5300
sudo iptables -D INPUT -p udp --dport 5300 -j ACCEPT
sudo netfilter-persistent save
```

Быстрая проверка: `ss -ulnp | grep ':53 '` (ожидается только `systemd-resolved` на `127.0.0.53`), `dig +short google.com @127.0.0.53` (локальный DNS работает), `iptables -t nat -L PREROUTING -n | grep 5300` (перенаправление активно).

### Управление сервисом

```bash
systemctl status thefeed-server
systemctl restart thefeed-server
journalctl -u thefeed-server -f

sudo vi /opt/thefeed/data/channels.txt   # отредактируйте каналы, затем:
sudo systemctl restart thefeed-server
```

Сервер также рисует терминальную панель из своих почасовых отчётов (`<data-dir>/dns_hourly.jsonl`) — ничего не отдаёт в сеть, только читает каталог данных:

```bash
thefeed-server --data-dir /srv/thefeed --report                   # снимок
thefeed-server --data-dir /srv/thefeed --report --report-refresh 5s   # в реальном времени
```

Показывает общее число запросов (fetch каналов / метаданные / медиа / чат), агрегаты по каналам и доменам и статистику чата.

<a id="messenger"></a>

## Мессенджер

**Необязательный** автономный store-and-forward мессенджер между пользователями одного сервера — он никак не связан с Telegram. Включается выделением одного или нескольких поддоменов (с соответствующей [NS-записью](#run-server), отдельных от доменов ленты):

```bash
thefeed-server ... --chat-domains c.example.com     # или THEFEED_CHAT_DOMAINS=c.example.com
```

- **Сквозное шифрование** — прочитать сообщение могут только двое; сервер хранит непрозрачные блобы и проверяет отправителя, ничего не читая. Имена контактов никогда не покидают устройство.
- **Идентичность** — клиент локально генерирует код восстановления; ваш адрес — 20 символов, выведенных из него. Передайте адрес по другому каналу; тот же код восстановления работает на любом сервере.
- **Fail-closed** — чат включается только если в профиле закреплён ключ сервера (`sk=`) и подписанная информация о поддержке чата проверена. Подписанный бит в метаданных ленты позволяет клиенту без ключа подсказать «на этом сервере есть мессенджер — переимпортируйте конфиг с его ключом» вместо тихого отказа.
- **Лимиты от злоупотреблений** (сообщаются клиентам): `--chat-send-per-hour` (30), `--chat-inbox-cap` (50), `--chat-per-pair-cap` (10), `--chat-max-msg-bytes` (500); недоставленные сообщения истекают через `--chat-ttl-hours` (72). `--chat-enabled=false` сохраняет домены, но объявляет чат отключённым.

В клиенте откройте **Chat** в нижней навигации. ✓ = сохранено на сервере, ✓✓ = получатель забрал; совпадающие эмодзи безопасности на обоих устройствах подтверждают, что разговор защищён.

<a id="how-it-works"></a>

## Как это работает

```
                                  Encrypted DNS TXT
   ┌──────────────┐  feed meta + small media   ┌──────────────────┐    MTProto      ┌──────────┐
   │              │ ─────────────────────────▸ │      Server      │ ─────────────▸  │ Telegram │
   │    Client    │ ◂───────────────────────── │  (DNS auth +     │ ◂─────────────  │   API    │
   │  (Web UI)    │                            │   media relays)  │    RSS / HTTP   ┌──────────┐
   │              │  large media (fast relay)  │                  │ ─────────────▸  │  Nitter  │
   │              │ ◂───── api.github.com ◂──  │                  │ ◂─────────────  │ (X feed) │
   └──────────────┘     (uploaded by server)   └──────────────────┘                 └──────────┘
```

**Сервер** (вне цензурируемой сети): подключается к Telegram и читает настроенные каналы; тянет публичные посты X через RSS-совместимые зеркала (без логина); отдаёт метаданные ленты и мелкое медиа как **зашифрованные DNS TXT**-ответы со случайным паддингом (анти-DPI). Вход один раз — работает всегда; `--no-telegram` читает публичные каналы без учётных данных. Необязательные режимы **мультидомена** и **[мессенджера](#messenger)**. Всё состояние — в одном каталоге данных.

**Клиент** (внутри цензурируемой сети): веб-интерфейс в браузере (RTL/фарси, шрифт VazirMatn), шлёт зашифрованные DNS-запросы через общий **банк резолверов** — резолверы находятся сканером, импортом или вручную и оцениваются по успешности + задержке, чтобы предпочитать здоровые. **Режим scatter** рассылает запрос нескольким резолверам и берёт самый быстрый ответ. Загрузка медиа учитывает релеи и проверяет хеш/размер. Включает [мессенджер](#messenger), автообновление по каналам, индикаторы новых сообщений и живой лог DNS-запросов.

### Протокол

Весь трафик зашифрован (AES-256) и передаётся в обычных DNS TXT-запросах/ответах с переменным паддингом и скорингом резолверов, сливаясь с нормальным DNS. Данные сообщений сжимаются (deflate) перед шифрованием; каждый запрос независим (без состояния сессии в канале).

<a id="media-relays"></a>

### Медиарелеи

Сообщения с фото, файлами, GIF, аудио и видео могут кешироваться на сервере и скачиваться по тому же каналу. Сервер дедуплицирует каждый файл (по id источника + хешу содержимого), проталкивает байты во все включённые релеи и добавляет к тексту сообщения небольшой заголовок:

```
[IMAGE]<size>:<flags>:<dnsCh>:<dnsBlk>:<crc32>[:<filename>]
optional caption
```

`<flags>` — список битов доступности по релеям через запятую (`1`=доступен, `0`=нет): слот 0 — DNS, слот 1 — GitHub, будущие релеи добавляются в конец; старые клиенты игнорируют незнакомые слоты. Каждый релей независим — один файл может отдаваться сразу несколькими путями. Клиент предпочитает самый быстрый, повторяет при сбое и спрашивает перед откатом на медленный путь. Блок 0 каждого файла в DNS-кеше начинается с 16-байтового заголовка (CRC32 + версия + байт компрессии + резерв), который клиент сверяет до выдачи байтов. Загрузки кешируются на клиенте (IndexedDB, 7 дней) и в локальном клиенте (`<dataDir>/media-cache/`, 7 дней).

Сейчас есть два релея:

- **DNS-релей** (медленный, устойчивый к цензуре, по умолчанию выключен) — байты разбиваются по DNS-блокам. Лимит по умолчанию 100 КБ.
- **GitHub-релей** (быстрый, по умолчанию выключен) — байты загружаются в репозиторий и забираются обычным HTTPS; нужен PAT с `contents:write`. Файлы ложатся в `<repo>/<sanitised-domain>/<size>_<crc32>`. Лимит по умолчанию 15 МБ.

| Flag | Env | Default | Notes |
|------|-----|---------|-------|
| `--dns-media-enabled` | `THEFEED_DNS_MEDIA_ENABLED` | `false` | вкл./выкл. DNS-релей |
| `--dns-media-max-size` | `THEFEED_DNS_MEDIA_MAX_SIZE_KB` | `100` КБ | лимит на файл |
| `--dns-media-compression` | `THEFEED_DNS_MEDIA_COMPRESSION` | `gzip` | `none` / `gzip` / `deflate` |
| `--github-relay-enabled` | `THEFEED_GITHUB_RELAY_ENABLED` | `false` | вкл./выкл. GitHub-релей |
| `--github-relay-token` | `THEFEED_GITHUB_RELAY_TOKEN` | — | PAT, `contents:write` |
| `--github-relay-repo` | `THEFEED_GITHUB_RELAY_REPO` | — | `owner/repo` |
| `--github-relay-branch` | `THEFEED_GITHUB_RELAY_BRANCH` | `main` | ветка |
| `--github-relay-max-size` | `THEFEED_GITHUB_RELAY_MAX_SIZE_KB` | `15360` КБ | лимит на файл |

<a id="security"></a>

## Безопасность

**Двухчастный контроль доступа:**

- **Пароль шифрования (`--key`)** — нужен и на сервере, и на клиенте. Любой, у кого он есть, может читать все сообщения каналов (включая приватные); делитесь только с доверенными.
- **Удалённое управление (`--allow-manage`)** — при включении любой с паролем может из клиента: **отправлять сообщения через Telegram-аккаунт сервера** (в каналы / личные чаты — это отправка через залогиненный аккаунт оператора, **полностью отдельно** от сквозного [мессенджера](#messenger)), и **менять список каналов ленты** (добавлять/удалять каналы, показываемые в ленте). По умолчанию выключено; включайте только на доверенных серверах.
- **Веб-пароль клиента (`--password`)** — HTTP Basic Auth на веб-интерфейсе. Только локальная защита; на доступ уровня DNS не влияет.

**Свойства:** сквозной AES-256 в обе стороны · случайный паддинг ломает анализ размеров · каждый запрос независим (без состояния сессии в канале) · запись за флагом `--allow-manage` · 2FA Telegram спрашивается интерактивно (никогда не хранится в аргументах) · файл session с правами `0600`.

> **⚠️ Никогда не публикуйте свой пароль.** Любой, у кого он есть, может запустить свой клиент и прочитать все ваши сообщения — предотвратить это нельзя. Флаг `--password` защищает только веб-интерфейс на вашей машине.

Необязательный [мессенджер](#messenger) шифруется сквозно отдельно для каждого разговора и независим от пароля ленты.

## Сборка из исходников

**Требования:** Go 1.26+ и (для приватных каналов) учётные данные Telegram API с <https://my.telegram.org>.

```bash
make build          # сборка сервера + клиента в ./build
make build-server   # только сервер
make build-client   # только клиент
make test           # тесты с race detector
make build-all      # кросс-компиляция всех платформ (вкл. Android)
make vet            # go vet
```

**Запуск сервера** (см. [флаги](#server-flags)):

```bash
./build/thefeed-server --data-dir ./data --domain t.example.com --key "passphrase" --no-telegram --listen ":5300"
# приватные каналы: один раз --login-only (с --api-id/--api-hash/--phone), потом запускайте без него
```

**Запуск клиента:** `./build/thefeed-client` (создаёт `./thefeeddata/`, открывает `http://127.0.0.1:8080`). Опции: `--data-dir`, `--port` (8080), `--password`. Опции каждого конфига (резолверы, scatter, rate limit, таймаут) задаются в веб-интерфейсе, а не флагами.

**macOS:** `make mac-dmg` → `build/Thefeed.app` + универсальный `.dmg`. Данные в `~/Library/Application Support/Thefeed`; пункт в строке меню даёт **Open / Quit**.

**Android APK:**

```bash
make build-android-arm64
cp build/thefeed-client-android-arm64 android/app/src/main/assets/thefeed-client
cd android && gradle wrapper --gradle-version 8.10.2 && ./gradlew assembleDebug
# → android/app/build/outputs/apk/debug/app-debug.apk
```

**iOS:** оборачивает Go-клиент в gomobile-xcframework, который использует SwiftUI-приложение в `ios/`. Сервер работает в процессе на `127.0.0.1:<random-port>`, только на переднем плане. Нужны Xcode 15+, Go 1.26+, gomobile (`go install golang.org/x/mobile/cmd/gomobile@latest && gomobile init`).

```bash
make ios-bind    # сборка Mobile.xcframework (устройство + симулятор)
make ios-build   # сборка приложения для симулятора
```

Затем откройте `ios/Thefeed.xcodeproj` в Xcode для запуска на устройстве.

**Релизы:** push тега, начинающегося с `v`, запускает CI-сборку + релиз. Тег с `-` (напр. `v1.4.0-rc1`) автоматически помечается как пре-релиз. Ассеты включают серверные/клиентские бинарники для всех платформ и Android APK (`arm64-v8a`, `armeabi-v7a`).

## Справочник

### Форматы файлов конфигурации

Все необязательные, лежат в каталоге данных; `#` начинает комментарий, пустые строки игнорируются.

- **`channels.txt`** — публичные Telegram-каналы, по одному `@username` в строке.
- **`x_accounts.txt`** — публичные имена пользователей X (Twitter), по одному в строке.
- **`private_channels.txt`** — ссылки-приглашения в приватные каналы, по одной в строке. Требует логина в Telegram — сервер при старте вступает в каждый канал и дальше читает его как обычный. Принимаются: `https://t.me/+…`, `t.me/joinchat/…`, `tg://join?invite=…` или голый хеш приглашения.

```
# channels.txt      # x_accounts.txt    # private_channels.txt
@VahidOnline        Vahid               https://t.me/+aBcDeF123456
```

### Веб-интерфейс

Оболочка в стиле Telegram с **нижней навигацией** из пяти разделов:

- **Feed** — лента каналов/X по типам (публичные/X/приватные), нативный RTL/фарси-рендер, плавающее поле ввода (отправка в каналы/личку при подключённом Telegram), бейджи новых сообщений по каналам, подсветка медиа-тегов, поиск по каналу и живой лог DNS. **Сохранённые сообщения** (зашифрованные локальные заметки + закладки) тоже здесь.
- **Mirror** (Telemirror) — доступное только для чтения зеркало каналов в стиле Telegram-web с рендером фото/альбомов, сохраняющим точное соотношение сторон.
- **Chat** — сквозно-зашифрованный [мессенджер](#messenger).
- **Resolver** — общий **банк**, ваши именованные **списки** резолверов и **сканер** в одном месте. Сканер прощупывает IP / CIDR (напр. `5.1.0.0/16`) или домены в поисках DNS-серверов, достающих до вашего сервера; результаты сортируются по задержке, расширяют /24 рабочего резолвера и применяются прямо к конфигу. Кнопка в одно нажатие загружает подобранный список CIDR.
- **Settings** — **Отображение** (тема, шрифт, язык, обои), **Соединение** (режим запроса, rate limit, scatter, таймаут, пароль, отладка), **Хранилище** (лимит дискового кеша), **Резервная копия** (зашифрованный экспорт/импорт), **О программе** и **Конфиги** (импорт/управление, с готовыми стартовыми конфигами). Тема по умолчанию следует за устройством. Все поля сохраняются автоматически.

### Безопасность получения X

Посты X берутся только через RSS/XML. URL инстансов валидируются (`http`/`https`, только хост), размер ответа ограничен и таймауты соблюдаются, при `403`/сбое сервер пробует следующий настроенный инстанс. Задайте свои доверенные зеркала через `--x-rss-instances`.
