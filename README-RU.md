# thefeed

Читалка Telegram-каналов и публичных аккаунтов X поверх DNS. Создана для сетей, где проходят только DNS-запросы.

[English](README.md) | [فارسی](README-FA.md) | [简体中文](README-ZH.md) | Русский

## Как это работает

```
                                  Encrypted DNS TXT
   ┌──────────────┐  feed meta + small media   ┌──────────────────┐    MTProto      ┌──────────┐
   │              │ ─────────────────────────▸ │      Server      │ ─────────────▸  │ Telegram │
   │    Client    │ ◂───────────────────────── │  (DNS auth +     │ ◂─────────────  │   API    │
   │  (Web UI)    │                            │   media relays)  │    RSS / HTTP   ┌──────────┐
   │              │  large media (fast relay)  │                  │ ─────────────▸  │  Nitter  │
   │              │ ◂───── api.github.com ◂──  │                  │ ◂─────────────  │ (X feed) │
   └──────────────┘     (uploaded by server)   └──────────────────┘                 └──────────┘
```

**Сервер** (работает за пределами цензурируемой сети):
- Подключается к Telegram, читает сообщения из настроенных каналов
- Получает публичные посты X через RSS-совместимые зеркала (без логина)
- Отдаёт метаданные ленты и небольшие медиафайлы в виде зашифрованных DNS TXT-ответов
- **Медиарелеи** — один и тот же файл, несколько путей доставки:
  - **DNS-релей** (медленный, устойчив к цензуре) разбивает байты на DNS-блоки
  - **GitHub-релей** (быстрый, по умолчанию выключен) загружает байты в репозиторий, клиент забирает их обычным HTTPS; для файлов, которые не влезают в DNS
  - Будущие релеи добавляются рядом, не ломая старые клиенты
- Случайный паддинг в ответах (анти-DPI)
- **Мультидомен** (`--extra-domains` / `THEFEED_EXTRA_DOMAINS`): сервер отвечает на feed-запросы на основном домене плюс на нескольких поддоменах, а клиенты распределяют запросы блоков по всем доменам (балансировка + устойчивость, если один домен фильтруется). Основной домен остаётся каноническим для путей релея. Поддомены передаются в поле `d=` URI импорта.
- **Мессенджер** (`--chat-domains` / `THEFEED_CHAT_DOMAINS`): необязательный store-and-forward мессенджер между пользователями одного сервера на выделенных поддоменах — см. раздел «Мессенджер»
- Постоянная сессия — один логин, работает дальше без вмешательства
- Режим без Telegram (`--no-telegram`) — публичные каналы читаются без учётных данных
- Все данные хранятся в одном каталоге

**Клиент** (работает внутри цензурируемой сети):
- Веб-интерфейс в браузере с поддержкой RTL/фарси (шрифт VazirMatn)
- Шлёт зашифрованные DNS TXT-запросы через банк резолверов
- **Банк резолверов**: общий пул DNS-резолверов для всех профилей. Резолверы добавляются сканером, импортом или вручную и автоматически оцениваются
- **Встроенные стартовые конфиги**: диалог импорта содержит готовые конфиги (с пресетом резолверов), импортируются в одно касание
- **Скоринг резолверов**: постоянная оценка каждого резолвера по успешности и задержке; здоровые резолверы в приоритете, низкорейтинговые можно вычистить
- **Режим scatter**: один и тот же DNS-запрос рассылается нескольким резолверам, используется самый быстрый ответ (по умолчанию 2 параллельных)
- **Загрузка медиа с учётом релеев** — выбирает быстрый релей, если он заявлен в манифесте, повторяет при сбоях, перед откатом на медленный DNS-путь спрашивает. Хеш и размер проверяются при каждой загрузке
- Отправка сообщений в каналы и личные чаты (нужен серверный `--allow-manage` + логин в Telegram)
- Управление каналами (удалённое добавление/удаление через админ-команды при включённом `--allow-manage`)
- **Автообновление по каналам**: закрепите нужные каналы для периодического фонового обновления, настройка хранится в профиле
- Сжатие сообщений (deflate) для эффективной передачи
- Парольная защита веб-интерфейса (`--password` у клиента)
- Индикаторы новых сообщений (бейдж NEW в списке каналов + разделитель в чате), таймер до следующей выборки
- Бейджи типов каналов (приватный/публичный/X) разными цветами
- Распознавание типов медиа (`[IMAGE]`, `[VIDEO]` и т.д.) и встроенный рендеринг
- Живой лог DNS-запросов прямо в браузере

## Протокол

Весь трафик зашифрован (AES-256) и передаётся в обычных DNS TXT-запросах/ответах; переменный паддинг и скоринг резолверов помогают трафику сливаться с нормальной DNS-активностью. Данные сообщений сжимаются (deflate) перед шифрованием.

## Загрузка изображений и файлов

Сообщения с фотографиями, файлами, GIF, аудио и видео могут кешироваться на сервере и скачиваться по тому же зашифрованному DNS-каналу.

Сервер скачивает каждый медиафайл (с дедупликацией по upstream id и хешу содержимого), отправляет байты во все включённые релеи и добавляет небольшой метазаголовок в текст сообщения:

```
[IMAGE]<size>:<flags>:<dnsCh>:<dnsBlk>:<crc32>[:<filename>]
optional caption
```

`<flags>` — список битов доступности по релеям через запятую (`1`=доступен, `0`=нет). Слот 0 — DNS, слот 1 — GitHub-релей; будущие релеи добавляются в конец. Старые клиенты игнорируют незнакомые слоты.

Блок 0 каждого файла в DNS-кеше начинается с 16-байтового протокольного заголовка: 4 байта CRC32 (от распакованного содержимого), 1 байт версии, 1 байт компрессии, 10 байт зарезервировано. Клиент сверяет CRC до выдачи байтов, остальное распаковывается согласно байту компрессии. Загрузки кешируются на клиенте (IndexedDB, 7 дней) и в локальном сервисе thefeed-client (`<dataDir>/media-cache/`, 7 дней). Число одновременных загрузок ограничено, лишние клики ставятся в очередь.

### Медиарелеи

Каждый релей независим — один и тот же файл может одновременно раздаваться через DNS, GitHub и будущие релеи. Клиент берёт то, что заявлено в манифесте сообщения, предпочитая самый быстрый доступный путь; при сбое повторяет, а перед откатом на более медленный путь спрашивает. Хеш и размер проверяются при каждой загрузке.

Сейчас есть два релея:

- **DNS-релей** (медленный, по умолчанию выключен). Байты разбиваются на DNS-блоки. Выживает в цензурируемых сетях. Лимит по умолчанию: 100 КБ.
- **GitHub-релей** (быстрый, по умолчанию выключен). Байты загружаются в репозиторий, клиент забирает их обычным HTTPS. Нужен personal access token с правом `contents:write`. Файлы ложатся в `<repo>/<sanitised-domain>/<size>_<crc32>`, так что несколько развёртываний могут делить один репозиторий. Лимит по умолчанию: 15 МБ.

Флаги сервера / переменные окружения:

| Флаг                          | Переменная                           | По умолчанию | Примечание                          |
|-------------------------------|--------------------------------------|-------------|-------------------------------------|
| `--dns-media-enabled`         | `THEFEED_DNS_MEDIA_ENABLED`          | `false`     | включить DNS-релей                  |
| `--dns-media-max-size`        | `THEFEED_DNS_MEDIA_MAX_SIZE_KB`      | `100` (KB)  | лимит на файл                       |
| `--dns-media-cache-ttl`       | `THEFEED_DNS_MEDIA_CACHE_TTL_MIN`    | `600` (min) | TTL                                 |
| `--dns-media-compression`     | `THEFEED_DNS_MEDIA_COMPRESSION`      | `gzip`      | `none`, `gzip` или `deflate`        |
| `--github-relay-enabled`      | `THEFEED_GITHUB_RELAY_ENABLED`       | `false`     | включить GitHub-релей               |
| `--github-relay-token`        | `THEFEED_GITHUB_RELAY_TOKEN`         | —           | PAT с правом `contents:write`       |
| `--github-relay-repo`         | `THEFEED_GITHUB_RELAY_REPO`          | —           | `owner/repo`                        |
| `--github-relay-branch`       | `THEFEED_GITHUB_RELAY_BRANCH`        | `main`      | ветка для объектов релея            |
| `--github-relay-max-size`     | `THEFEED_GITHUB_RELAY_MAX_SIZE_KB`   | `15360` (KB)| лимит на файл                       |
| `--github-relay-ttl`          | `THEFEED_GITHUB_RELAY_TTL_MIN`       | `600` (min) | осиротевшие объекты чистятся в следующем цикле |

Ежечасный DNS-отчёт включает `totalMediaQueries` и блок `mediaCache` (записи, байты, попадания, промахи, вытеснения).

## Мессенджер

Необязательный автономный store-and-forward мессенджер между пользователями одного сервера (никак не связан с Telegram). Включается выделенными поддоменами, отличными от feed-доменов:

```bash
thefeed-server ... --chat-domains c.example.com
# или: THEFEED_CHAT_DOMAINS=c.example.com
```

- **Сквозное шифрование**: сообщение могут прочитать только двое — сервер хранит непрозрачные блобы и проверяет отправителя, не читая содержимое. Имена контактов никогда не покидают устройство.
- **Идентичность**: клиент локально создаёт код восстановления; ваш адрес — 20 символов, производных от него. Передайте адрес по другому каналу, чтобы вам могли писать; один и тот же код восстановления работает на любом сервере.
- **Fail-closed безопасность**: клиент включает чат, только если в профиле закреплён ключ сервера (`sk=`) и данные о возможностях чата проходят проверку подписи. Подписанный бит в метаданных feed сообщает, что у сервера есть мессенджер, поэтому клиент *без* ключа может сказать «у этого сервера есть мессенджер — переимпортируйте конфиг с его ключом» вместо тихого отказа, а на серверах без чата клиенты не тратят лишний запрос.
- **Выключение без удаления доменов**: `--chat-enabled=false` (или `THEFEED_CHAT_ENABLED=0`) сохраняет домены, но объявляет чат отключённым; клиенты показывают «мессенджер отключён этим сервером».
- **Лимиты против злоупотреблений** (автоматически сообщаются клиентам): `--chat-send-per-hour` (30), `--chat-inbox-cap` (50), `--chat-per-pair-cap` (10), `--chat-max-msg-bytes` (500). Недоставленные сообщения истекают через `--chat-ttl-hours` (72).
- **Аккаунты по умолчанию сохраняются** (`--chat-account-ttl-days 0`), чтобы отчёты были точными; на нагруженном сервере задайте число дней, чтобы освобождать неактивные аккаунты. `--chat-max-accounts` (0 = без лимита) ограничивает общее число аккаунтов.
- **Надёжность против пропускной способности** (`--chat-sync-seconds`, по умолчанию 1): хранилище сообщений сбрасывается на диск каждые N секунд, поэтому при сбое можно потерять до ~N секунд только что полученных сообщений (не страшно — чат сквозной, отправители повторят). `0` = fsync на каждое сообщение (строгая надёжность, ниже пропускная способность).
- В интерфейсе клиента откройте **Мессенджер** в боковой панели. ✓ = сохранено на сервере, ✓✓ = получено адресатом. Совпадающие защитные эмодзи на обоих устройствах подтверждают безопасность переписки.

Часовой DNS-отчёт включает `totalChatQueries` и блок `chat` (аккаунты, сообщения, регистрации, сессии).

### Отчёт оператора (TUI)

Сервер дописывает каждый часовой отчёт одной строкой JSON в `<data-dir>/dns_hourly.jsonl` (ротация по размеру, несколько резервных копий). Сам бинарник сервера рисует терминальную панель из этого файла по флагу `--report`; он ничего не публикует в сеть, только читает каталог данных:

```bash
thefeed-server --report                       # читает ./data/dns_hourly.jsonl и ./data/chat.db
thefeed-server --data-dir /srv/thefeed --report
thefeed-server --report --report-refresh 5s   # вживую, перерисовка каждые 5 с
thefeed-server --report --report-from 2026-06-10 --report-to "2026-06-11 18:00"   # только этот диапазон UTC
```

Показывает суммарные запросы / channel-fetch / метаданные / медиа / чат, средние по каналам со столбчатыми диаграммами, суммы по доменам, спарклайн по отчётам, запросы по часам суток и статистику чата (включая живой счётчик аккаунтов из `chat.db`) — те же агрегаты, что и `scripts/thefeed_log_report.py`, в терминале.

## Быстрая установка (сервер)

Установщик умеет тянуть бинарники с зеркала GitHub или GitLab.
По умолчанию определяет сам (сначала пробует GitHub, при неудаче — GitLab);
флаг `--gitlab` принудительно использует GitLab, если аккаунт GitHub недоступен.

```bash
# Зеркало GitHub
sudo bash -c "$(curl -Ls https://raw.githubusercontent.com/sartoopjj/thefeed/main/scripts/install.sh)"

# Зеркало GitLab
sudo bash -c "$(curl -Ls https://gitlab.com/sartoopjj/thefeed/-/raw/main/scripts/install.sh)" -- --gitlab
```

Или вручную:

```bash
# На вашем сервере (Linux с systemd)
curl -Ls https://raw.githubusercontent.com/sartoopjj/thefeed/main/scripts/install.sh -o install.sh
sudo bash install.sh                # авто: GitHub, затем откат на GitLab
sudo bash install.sh --gitlab       # принудительно зеркало GitLab
sudo bash install.sh --source github
```

Скрипт:
1. Скачает бинарник последнего релиза с GitHub
2. Спросит домен, пароль-фразу, Telegram-каналы и аккаунты X
3. Спросит, использовать ли логин в Telegram (рекомендуется **Нет** — публичные каналы работают без него)
4. В режиме Telegram: спросит API-учётные данные и выполнит вход
5. Настроит systemd-сервис

Обновление:
```bash
sudo bash -c "$(curl -Ls https://raw.githubusercontent.com/sartoopjj/thefeed/main/scripts/install.sh)"
```

Установка конкретной версии (откат, beta или rc):
```bash
# Откат на проверенный тег
curl -Ls https://raw.githubusercontent.com/sartoopjj/thefeed/main/scripts/install.sh | sudo bash -s -- --version v0.9.2

# Установка свежайшего пре-релиза (beta / rc)
curl -Ls https://raw.githubusercontent.com/sartoopjj/thefeed/main/scripts/install.sh | sudo bash -s -- --pre

# Список последних релизов (stable / pre-release)
curl -Ls https://raw.githubusercontent.com/sartoopjj/thefeed/main/scripts/install.sh | sudo bash -s -- --list
```

Короткие формы: `-v <tag>` — то же, что `--version <tag>`. Старая позиционная форма
`sudo bash install.sh v1.0.0` тоже работает.

Повторный вход: `curl -Ls https://raw.githubusercontent.com/sartoopjj/thefeed/main/scripts/install.sh | sudo bash -s -- --login`
Показать конфиг (URI для импорта — домен, ключ, серверный ключ `sk=`, резолверы): `curl -Ls https://raw.githubusercontent.com/sartoopjj/thefeed/main/scripts/install.sh | sudo bash -s -- --config`
Удаление: `curl -Ls https://raw.githubusercontent.com/sartoopjj/thefeed/main/scripts/install.sh | sudo bash -s -- --uninstall`


> **Примечание:** сервер должен принимать пакеты на внешнем порту 53. Слушать `:53` напрямую можно только под root. Лучше слушать непривилегированный порт (`:5300`) и пробросить на него порт 53.
>
> Замените `eth0` на имя вашего сетевого интерфейса (смотрите `ip a`):
> ```bash
> sudo iptables -I INPUT -p udp --dport 5300 -j ACCEPT
> sudo iptables -t nat -I PREROUTING -i eth0 -p udp --dport 53 -j REDIRECT --to-ports 5300
> sudo ip6tables -I INPUT -p udp --dport 5300 -j ACCEPT
> sudo ip6tables -t nat -I PREROUTING -i eth0 -p udp --dport 53 -j REDIRECT --to-ports 5300
> ```
>
> Чтобы правила пережили перезагрузку:
> ```bash
> sudo apt install iptables-persistent   # Debian/Ubuntu
> sudo netfilter-persistent save
> ```



**Если что-то пошло не так — мгновенно убрать редирект:**

```bash
# Удаляет правило iptables (возвращает исходное поведение)
sudo iptables -t nat -D PREROUTING -i eth0 -p udp --dport 53 -j REDIRECT --to-ports 5300
sudo iptables -D INPUT -p udp --dport 5300 -j ACCEPT
sudo netfilter-persistent save
```

## Развёртывание в Docker (сервер)

Запуск сервера в Docker — Go-инструментарий не нужен.

### Быстрый старт (публичные каналы, без логина в Telegram)

```bash
# 1. Настроить окружение
cp .env.example .env
nano .env   # задать THEFEED_DOMAIN и THEFEED_KEY

# 2. Подготовить каталог данных со списком каналов
mkdir -p data
cp configs/channels.txt data/
cp configs/x_accounts.txt data/   # опционально

# 3. Собрать и запустить
docker compose up -d

# 4. Перенаправить внешний DNS-трафик в контейнер
#    Замените eth0 на ваш интерфейс (смотрите: ip a)
sudo iptables -t nat -I PREROUTING -i eth0 -p udp --dport 53 -j REDIRECT --to-ports 5300
sudo iptables -I INPUT -p udp --dport 5300 -j ACCEPT
sudo ip6tables -t nat -I PREROUTING -i eth0 -p udp --dport 53 -j REDIRECT --to-ports 5300
sudo ip6tables -I INPUT -p udp --dport 5300 -j ACCEPT

# Сделать правила iptables постоянными
sudo apt install -y iptables-persistent
sudo netfilter-persistent save

# 5. Смотреть логи
docker compose logs -f
```

> **Примечание:** контейнер слушает порт 5300 (не 53), чтобы не конфликтовать с `systemd-resolved`.
> Правило `iptables PREROUTING` перенаправляет в контейнер только **внешний** DNS-трафик (порт 53),
> локальное разрешение имён на сервере продолжает работать как обычно.

### Порт 53 и безопасность сервисов

Контейнер слушает порт **5300** (не 53), чтобы не конфликтовать с `systemd-resolved` и другими DNS-сервисами на хосте. Внешний DNS-трафик перенаправляется через `iptables PREROUTING`, что затрагивает только пакеты на внешнем интерфейсе — локальное разрешение имён **не** страдает.

**До настройки — проверьте, кто занимает порт 53:**

```bash
# Занят ли порт 53
ss -ulnp | grep ':53 '

# Ожидаемо: только systemd-resolved на 127.0.0.53 (безопасно)
# UNCONN  127.0.0.53%lo:53  users:(("systemd-resolve",...))
```

**После настройки — убедитесь, что ничего не сломалось:**

```bash
# 1. Локальный DNS работает (сервер разрешает домены)
dig +short google.com @127.0.0.53

# 2. Контейнер thefeed запущен
docker ps --filter name=thefeed

# 3. thefeed забирает каналы
docker logs thefeed-server --tail 5

# 4. Правило iptables активно
iptables -t nat -L PREROUTING -n | grep 5300

# 5. Остальные контейнеры в порядке
docker ps --format 'table {{.Names}}\t{{.Status}}' | head -10
```

**Если что-то пошло не так — мгновенно убрать редирект:**

```bash
# Удаляет правило iptables (возвращает исходное поведение)
sudo iptables -t nat -D PREROUTING -i eth0 -p udp --dport 53 -j REDIRECT --to-ports 5300
sudo iptables -D INPUT -p udp --dport 5300 -j ACCEPT
sudo netfilter-persistent save
```

### Веб-интерфейс

В браузерном интерфейсе есть:
- **Боковая панель каналов** (слева): список каналов по типам (публичные/X/приватные) с бейджами
- **Панель сообщений** (справа): сообщения с нативным RTL/фарси-рендерингом (шрифт VazirMatn)
- **Панель отправки**: отправка сообщений в каналы и личные чаты при подключённом Telegram
- **Бейджи новых сообщений**: видно, в каких каналах есть новое
- **Таймер следующей выборки**: обратный отсчёт до автообновления
- **Распознавание медиа**: подсветка тегов `[IMAGE]`, `[VIDEO]`, `[DOCUMENT]`
- **Поиск по сообщениям**: поиск в текущем канале с подсветкой совпадений и навигацией вперёд/назад
- **Экспорт сообщений**: экспорт последних N сообщений канала в буфер обмена
- **Панель лога** (внизу): живой лог DNS-запросов
- **Окно настроек**: домен, пароль-фраза, резолверы, режим запросов, лимит скорости, параллельные запросы (scatter), таймаут, режим отладки
- **Рабочие резолверы**: список активных/здоровых резолверов в настройках
- **Фоновое изображение**: свой URL фоновой картинки для панели сообщений (хранится локально)
- **Таймаут DNS-запросов**: настраивается на профиль (по умолчанию 15s) в редакторе профиля
- **Кеш на профиль**: часовой кеш в браузере, данные видны мгновенно при повторном открытии
- **Сканер резолверов**: сканирование IP-диапазонов и CIDR для поиска рабочих DNS-резолверов

### Сканер резолверов

В веб-интерфейс встроен сканер резолверов (иконка 🔍 в боковой панели), который прощупывает IP-диапазоны в поисках DNS-серверов, способных достучаться до вашего сервера thefeed. Возможности:

- **Гибкие цели**: отдельные IP, CIDR (например `5.1.0.0/16`) или доменные имена — по одному в строке
- **Преднабор CIDR**: кнопка загрузки встроенного списка отобранных диапазонов
- **Очистка целей**: кнопка быстрой очистки списка CIDR/IP
- **Учёт профиля**: выбор, чей домен и пароль-фразу использовать для проб
- **Настраиваемость**: параллельность (по умолчанию 50), таймаут (по умолчанию 15s), максимум сканируемых IP
- **Расширение /24**: при нахождении рабочего резолвера автоматически сканируются соседние IP той же /24-подсети
- **Пауза / Продолжить / Стоп**: полный контроль над долгими сканированиями (пауза действительно останавливает новые пробы)
- **Время ответа**: результаты сортируются по задержке, самые быстрые сверху
- **Выбор результатов**: чекбоксы для выбора резолверов для применения или копирования
- **Применение результатов**: добавить к списку резолверов профиля или заменить его прямо из сканера
- **Копирование**: по одному IP, выбранные или все найденные
- **Новое сканирование**: сброс интерфейса для следующего прохода
- **Отладочный лог**: при включённой отладке логируется каждая проба (запрос/ответ)
- **Быстрый вход из редактора профиля**: кнопка "Find Resolvers" на странице редактирования профиля

## Безопасность

### Двухуровневый контроль доступа

**Пароль-фраза шифрования (`--key`):** нужна и серверу, и клиенту. Любой, у кого она есть, может читать все сообщения каналов (включая приватные). Можно поделиться с доверенными друзьями, чтобы читали вместе.

**Удалённое управление (`--allow-manage` на сервере):** когда включено, обладатель ключа шифрования может также отправлять сообщения и управлять каналами. По умолчанию выключено. Включайте только на доверенных серверах.

**Веб-пароль клиента (`--password`):** защищает все эндпоинты веб-интерфейса HTTP Basic Auth. Это только **локальная** защита — на доступ уровня DNS она не влияет.

### Свойства безопасности

- Весь трафик зашифрован сквозным шифрованием (AES-256)
- Обе стороны требуют заранее согласованную пароль-фразу
- Каждый запрос независим — в канале нет состояния сессии
- Случайный паддинг в обе стороны мешает анализу трафика
- Операции записи ограничены серверным флагом `--allow-manage`
- Пароль 2FA Telegram вводится интерактивно (никогда не хранится в аргументах)
- Файл сессии хранится с ограниченными правами (0600)

> **⚠️ Внимание:** если вы опубликуете пароль-фразу, **кто угодно** сможет запустить свой
> клиент с ней и читать все ваши сообщения. Помешать этому невозможно.
> Флаг клиента `--password` защищает только веб-интерфейс на вашей машине — он НЕ мешает
> другим пользоваться пароль-фразой. **Никогда не публикуйте пароль-фразу.**
